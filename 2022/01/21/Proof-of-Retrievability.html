Quick introduction to Proof of Retrievability

Because data is becoming more personal and is taking more storage space, it is quite usual now to load a file in Google Drive, OneDrive from Microsoft, etc.

Maybe, some of this data is important for you and you want a way to make sure that data is truly stored.

Unless you want to download entierely this data everyday, there is no solution for you to check if it’s stored.

Or maybe there is in fact one way: Proof of Retrievability.

To sum up, it’s a cryptographic tool enabling a user to mathematically challenge a server to check that its file is still stored.

It works in a five-step protocol involving three parties:

  • Client: the one that possesses the file;
  • Server: the one that has a lot of free storage space to rent;
  • Other: other user of the network.

The steps are:

  1. Initialization: Client generates some metadatas to start the process;
  2. Signature: Client signs its original file, making a new signature file. Client sends both files (original and signature) to Server;
  3. Challenge: Client sends a challenge to the Server;
  4. Proof: Server computes a proof according to the challenge, the original file and the signature file;
  5. Verification: Other or Client verify that the proof is correct regarding metadatas and challenge.

That’s it! The proof is way lighter than the file and easy to verify!

Work done

With the blockchain emergence, Proof of Retrievability found a place in this system.

Indeed, it just needs a smart contract that handle the verification step and here we go, we have a trusted system and a full protocol to store our file!

That’s why I worked with ISAE Supaéro on it and creates BlockHouse.

I am currently continuing this work with ISAE Supaéro and improving this protocol.

You can find a small part in the Github repository.

Versions

Version 1

Version state: Finished

The main idea is to make the public Compact Proof of Retrievability work. You can find it here: Compact Proofs of Retrievability

The smart contract stores all the require metadatas to verify proof, the challenges and the proofs.

A little test script is implemented to verify the protocol.

To sum up:

  • Public Compact Proof of Retrievability protocol in smart contract;
  • Test script to check if it works.

Version 2

Version state: Pending

There, I want to share a tool made from libff library to make all the steps describe above.

Then, anybody in a blockchain network with this solution will be able to run entirely this protocol.

I want to add benchmark tool in order to optimize the signing step. Indeed, signing a file generates new elements (to store aswell), so the lighter the better, otherwise this solution requires more storage space than at the start.

To sum up:

  • C++ tool to sign file, generate challenge, compute proofs and, incidentally, verify the proof;
  • Benchmark tool to optimize the signing step and display the protocol performances.

Version 3

Version state: Not started yet

Once everything is done with the protocol by itself, I want to make a marketplace where any account can propose some storage space, or a file to store with the possibility to check there reliability.

Payment will be made thanks to tokens, not only ether transfers.

I will create a site to display all propositions and a way to make challenges easier.

To sum up:

  • Marketplace smart contract handling all PoR protocols;
  • Website that displays the marketplace and facilitate usage of every protocol;
  • Payment by tokens.
